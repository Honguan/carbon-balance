# 標準、法規、查驗與專業認證

> 本文件是工程規劃摘要，不是法律或查驗意見。正式申請應由合格領域人員及環境部許可查驗機構確認。

## 1. 核心國際標準

| 文件 | 專案用途 | 系統需要支援的概念 |
|---|---|---|
| ISO 14067:2018 | 產品碳足跡量化與報告 | 目標與範疇、系統邊界、功能／宣告單位、LCI、分配、數據品質、結果與報告 |
| ISO 14040:2006 + Amd 1:2020 | LCA 原則與框架 | Goal & Scope、LCI、LCIA、Interpretation、報告、critical review |
| ISO 14044:2006 + Amd 1:2017 + Amd 2:2020 | LCA 要求與指引 | 資料、分配、比較、品質、限制、報告、審查要求 |
| ISO/TS 14027:2017 | PCR 制定、審查、登錄與更新 | 產品類別、PCR 版本、利害關係人、可比較性、更新與有效性 |

ISO 14067:2018 目前仍是已發布版本，但 ISO 官方已標示進入修訂流程。因此系統不得把標準條文硬編碼成不可變邏輯；需保存 `standard_id + edition + amendment + effective_date`，並允許規則版本並存。

## 2. 台灣現行申請要求

依 2025-03-19 發布之《自願性產品碳足跡核定標示及管理辦法》，申請產品碳足跡核定時，核心文件包括：

1. 申請者、產品與生產廠場／服務場所基本資料。
2. 公司、商業或主管機關登記證明。
3. 製造廠合法設立／登記證明（依法免辦者除外）。
4. 標籤使用與標示方式。
5. 產品生命週期流程圖。
6. 產品碳足跡盤查清冊與盤查報告書。
7. 環境部許可查驗機構出具之查驗聲明書。
8. 主管機關指定的其他資料。

盤查需符合：

- 最新發行之 ISO 14067。
- 主管機關公開且適用的 PCR。
- 活動資料原則上為申請日前二年內。
- 揭露同廠產品產量與共用能資源分配方式。
- 係數來源符合主管機關、國際文獻／LCA 資料庫、檢測報告或依 ISO 14067 計算之自廠係數等規則。

產品碳足跡核定有效期間為二年；申請相關資料依法至少保存三年。系統應以設定保留政策實作，且不得因 UI 刪除而破壞法定保存或查驗證據。

## 3. PCR 使用規則

每一個盤查專案至少保存：

- PCR 登錄編號、名稱、版本、來源機關。
- CCC Code／服務分類與適用性判斷。
- 核准日期、有效期間、狀態與下載來源。
- 原始 PDF checksum。
- 功能／標示單位。
- 系統邊界與生命週期階段。
- 一級／二級數據規則。
- 切斷準則。
- 分配規則。
- 資料品質與報告要求。
- 不適用階段與排除項目。
- 專案套用時的人工判斷、理由與審核人。

雲端的 `20-058 瓶裝水第5.0版` 文件載明核准日為 2021-02-08 且有效期五年。依此推算，2026 年不能直接把它當成現行有效 PCR；實作與測試可保留為歷史案例，但正式專案必須即時查詢主管機關的有效清單。

## 4. 排放係數治理

2026-01-16 環境部另公告《碳足跡排放係數建置及使用作業指引（第一版）》。係數模組應至少具備：

- 來源機關／資料庫／文獻。
- 原始名稱、標準化名稱與多語名稱。
- 數值與原始精度。
- 分子／分母單位與可換算維度。
- 地理、時間、技術與生命週期範疇。
- 公告／建立／有效起訖日期。
- 第三方查驗或資料品質資訊。
- 授權與使用限制。
- 匯入批次、原始檔、checksum、建立者與審核者。
- superseded／withdrawn 狀態。

更新係數不得改寫既有 calculation run。

## 5. 專業認證：什麼是必要、什麼不是

### 軟體開發團隊

建立與維護軟體本身，現行規則沒有要求每位開發者必須持有產品碳足跡查驗員資格。開發者不能因此對外簽發查驗聲明。

### 正式計算與申請

申請者可自行計算或委由輔導公司協助，但盤查清冊與報告必須由環境部許可之查驗機構查驗，並由其出具查驗聲明書。

### 執行查驗的人員與機構

《溫室氣體認證機構及查驗機構管理辦法》對查驗機構許可、查驗人員學經歷、訓練、實績及產品碳足跡主導查驗員有明確要求。這些資格屬於「查驗業務」，不能由一般軟體公司或系統自動取代。

### 建議團隊配置

| 角色 | 最低建議能力 | 是否為開發法定證照 |
|---|---|---|
| 產品負責人 | 產品碳足跡與 LCA 流程 | 否 |
| 領域審核者 | 熟悉 ISO 14067、PCR、盤查報告 | 否；但需具備可證明能力 |
| 查驗聯絡窗口 | 熟悉查驗機構要求與補正流程 | 否 |
| 軟體工程師 | 安全、資料、測試與稽核系統 | 否 |
| 正式查驗員／主導查驗員 | 符合法規訓練與實績 | 是，僅在執行查驗時適用 |
| 許可查驗機構 | 主管機關許可與認證 | 是，出具查驗聲明時必要 |

可安排 ISO 14067／LCA 相關訓練作為團隊能力證明，但訓練證書不得包裝成官方查驗資格。

## 6. 系統輸出狀態

報告與 UI 應明確區分：

```text
Draft                 草稿，尚未內部審核
Internally Reviewed   已內部審核，尚未第三方查驗
Submitted for Review  已送交查驗
Verified              已取得有效查驗聲明（需保存聲明資料）
Approved               已取得主管機關核定（需保存核定編號與期限）
Expired                查驗／核定／PCR 已逾期
Superseded             已由新版本取代
```

任何 `Verified` 或 `Approved` 狀態必須由有權限者輸入證明文件、編號、機構、日期與期限，不得依計算成功自動產生。

## 7. 文件與證據保存

系統應保存：

- 每一筆活動數據的來源、期間、供應者、上傳文件與審核狀態。
- 係數原始資料與版本。
- PCR／標準版本。
- 分配、排除、估算與替代數據理由。
- 計算公式版本與完整明細。
- 審核、核准、變更與匯出事件。
- 查驗發現、補正紀錄、聲明書與核定資料。

保留時間：一般申請資料至少三年；若系統提供查驗機構工作流程，查驗相關紀錄需依該機構適用規則配置更長期限，規劃上以六年以上並可依法規調整較安全。
