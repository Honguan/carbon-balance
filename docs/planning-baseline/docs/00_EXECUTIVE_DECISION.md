# 執行決策：重構或重新開始

## 結論

**正式版應重新開始，不建議在現有 PHP 專題上做漸進式重構。**

這不是否定舊專題的研究成果；舊專題的價值主要在：

- 問題定義與使用情境。
- 五階段盤查流程。
- 37 個功能的需求線索。
- 共通架構符號與公式概念。
- 碳係數資料、PCR、案例試算表與展示資料。
- UCD、事件表、情節描述、報告與論文。

但程式碼本身不具備可安全延伸為正式多使用者系統的基礎。

## 為何不是直接重構

### 1. 安全基礎必須全面替換

現有登入、帳號啟用、密碼、SQL、Session 與權限模型都有高風險問題。修補這些問題會連帶改動幾乎所有頁面、資料存取與流程，已超過一般重構的合理範圍。

### 2. 資料模型不能支持稽核與重現

現有資料庫混用 MyISAM／InnoDB，數值與日期大量以 `varchar` 儲存，缺少穩定關聯、版本、來源、有效期間、公式版本、計算快照與稽核紀錄。正式產品必須從資料模型重新設計。

### 3. 程式是頁面導向原型，不是可測試的領域系統

查詢、計算、HTML、寫入與導頁混在同一 PHP 檔；部分頁面在顯示時直接更新計算結果；沒有可獨立測試的計算核心。

### 4. 舊功能完成度不一致

專案內仍有 hard-coded 展示資料、測試頁、複製檔、Python 試驗檔、不同資料庫名稱與購物網站殘留程式。這代表現況是多個局部原型的集合。

### 5. 缺少交付工程

沒有正式 dependency manifest、可重建環境、migration、測試框架、CI、release、secret management、觀測性或災難復原方案。

## 建議的保存方式

- 舊版 repository 設為唯讀或建立 `legacy-final` tag。
- 不把舊版資料夾直接放進新方案並持續編譯／部署。
- 需要對照時，以 `legacy-reference` branch 或獨立私有 repository 保存。
- 將可用資產逐項提煉成：需求、資料字典、Golden Case、匯入檔與 UI 參考圖。

## 可保留與不可直接保留

| 類別 | 處置 |
|---|---|
| 專題報告、論文、UCD、事件表 | 保留，轉成現行需求與測試案例 |
| PCR、法規、係數資料 | 保留來源，但重新確認版本與有效性 |
| Excel 案例 | 保留，轉成 Golden Test；先核對公式與單位 |
| 公式概念 | 保留，需由領域人員審核並版本化 |
| 舊 SQL schema | 僅供資料映射參考，不沿用 |
| PHP 頁面與 SQL | 不直接移植；必要 UX 可重新實作 |
| 登入、帳號啟用、管理員判斷 | 全面捨棄 |
| Google Maps API Key | 立即撤銷／限制，禁止移植 |
| 舊資料庫內容 | 先進 staging、清洗與驗證，不直接匯入正式庫 |

## 立即處理事項

1. 撤銷或限制 repository 中已暴露的 Google Maps API Key。
2. 確認舊系統是否仍公開；若公開，先下線登入與寫入功能或限制存取。
3. 對舊資料庫做唯讀備份與 checksum。
4. 整理有效 PCR、係數來源、公式與案例，不再以檔名中的「最新」判定權威性。
5. 建立新 repository，從架構骨架與 Golden Case 開始。
