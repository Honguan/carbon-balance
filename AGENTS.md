# AGENTS.md

## 專案目標

建立可維護、可稽核、可重現的產品碳足跡盤查與計算系統。所有實作必須同時滿足：

- 正確表達產品生命週期、PCR、盤查資料、排放係數、分配與計算關係。
- 保存完整來源、版本、證據與稽核軌跡。
- 不把舊版資料或公式默認為正確；必須經驗證後才匯入正式模型。
- 優先採用目前受支援、易維護的版本與標準功能，避免不必要框架與微服務。

## 權威文件順序

遇到衝突時，依下列順序處理：

1. 現行法律、主管機關規則與已確認有效的 PCR。
2. ISO 14067、ISO 14040、ISO 14044、ISO/TS 14027 的適用要求。
3. 已核准 ADR。
4. `docs/04_PRODUCT_REQUIREMENTS.md` 與領域規則文件。
5. GitHub Issue／驗收標準。
6. 舊專題報告、試算表與舊程式碼。

舊程式碼只提供歷史線索，不是規格來源。

## 行動與核准邊界

- 對「分析、檢查、說明、規劃」要求：讀取必要資料並回報，不修改程式。
- 對「建立、修正、實作」要求：可直接進行範圍內的本機修改，並執行非破壞性驗證。
- 下列操作必須先取得明確核准：外部寫入、部署正式環境、刪除資料、旋轉正式憑證、改動正式資料庫、產生費用、擴大需求範圍。
- 發現法律、PCR、公式或資料語意有重大歧義時，停止該項領域決策並提出問題；其餘不受影響的工作繼續進行。

## 開發原則

- 使用 C#、ASP.NET Core 10 LTS、PostgreSQL 18。
- 採模組化單體；不得在沒有量測證據前拆成微服務。
- Domain 層不得依賴 Web、資料庫、檔案系統或外部 API。
- 所有資料表使用穩定 ID；名稱、地址、產品名稱不可作為關聯鍵。
- 碳排、活動量、係數與比例使用 `decimal`／PostgreSQL `numeric`，並保存原始精度。
- 單位不可只存文字；使用受控單位代碼與明確換算規則。
- 排放係數、PCR、公式與 GWP 必須版本化。
- Calculation Run 是不可變快照；修正資料後建立新版本，不覆寫歷史結果。
- 所有外部輸入在邊界驗證；所有 SQL 參數化。
- 密碼使用框架安全機制；不得自行設計密碼雜湊或驗證流程。
- 不在畫面讀取、GET 請求或報表查詢時偷偷更新資料。
- 不提交 secrets、實際客戶資料、受著作權保護的完整 ISO 內容。

## 實作流程

1. 讀取 Issue、相關需求、ADR 與領域規則。
2. 列出會變更的模組、資料結構、風險與驗收條件。
3. 進行最小且完整的垂直切片實作。
4. 新增或更新測試：Domain unit、integration、authorization、migration、Golden calculation。
5. 執行格式化、編譯、測試、靜態分析與 migration 驗證。
6. 更新受影響文件與 ADR。
7. 回報完成內容、驗證證據、未解決風險；不要以「應該可行」取代實際驗證。

## 必要驗證

一般變更至少通過：

```text
dotnet restore
dotnet build --configuration Release --no-restore
dotnet test --configuration Release --no-build
```

涉及資料庫時另需：

```text
建立全新資料庫並套用所有 migration
由前一個正式版本升級至目前版本
執行 migration rollback/recovery 演練（依變更風險）
```

涉及計算時另需：

- 至少一組手算或受控試算表 Golden Case。
- 驗證單位換算、四捨五入位置、分配、切斷、負值與零值。
- 驗證結果明細總和與階段／產品總和一致。
- 驗證舊 calculation run 不因係數更新而改變。

## 完成回報格式

1. 結論。
2. 修改檔案與模組。
3. 已執行的驗證及結果。
4. 資料／相容性／安全影響。
5. 尚未完成或需領域確認的項目。

## 禁止事項

- 不得為了通過測試而刪除或弱化測試。
- 不得把資料庫錯誤、轉換錯誤或計算錯誤吞掉後回傳成功。
- 不得以 UI 隱藏取代後端授權。
- 不得直接複製舊版登入、SQL、資料表或 Google Maps Key。
- 不得聲稱系統、報告或計算「通過 ISO 認證」；除非有明確、有效且適用的證明。
